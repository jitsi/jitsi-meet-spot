# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

`spot-controller` is the **Expo (SDK 56, managed workflow)** React Native (**0.85, React 19**) iOS/Android app for the dedicated in-room Spot-Remote. It does not reimplement the remote UI: it loads the web Spot-Remote in a `react-native-webview` and layers native-only capabilities (keep-awake, a side drawer, deployment switching, a JS<->native message bridge) around it. It is written in **strict TypeScript** (`.ts`/`.tsx`) and is an npm-workspaces member. Run the npm scripts from inside `spot-controller/` (or from the repo root with `--workspace spot-controller`).

## Commands

```bash
npm run lint        # eslint . (flat config)
npm run lint:fix    # eslint . --fix
npm run typecheck   # tsc --noEmit -p tsconfig.json (strict)
npm test            # jest (ts-jest) — unit tests for the React-Native-independent modules

npm start           # expo start (Metro dev server)
npm run prebuild    # expo prebuild --clean (regenerate ios/ + android/ from app.config.js)
npm run ios         # expo run:ios     (prebuild if needed, pod install, build+run on a simulator)
npm run android     # expo run:android (prebuild if needed, gradle build+run on an emulator/device)
```

Metro is the bundler (no `tsc` build step); type-checking is `npm run typecheck`. Dep versions are pinned to **Expo SDK 56** — use `npx expo install <pkg>` (not bare `npm install <pkg>`) so versions stay SDK-compatible, and `npx expo install --check` to audit.

**Testing scope:** jest + `ts-jest` run in a Node env (`tsconfig.spec.json`, CommonJS) and cover only the React-Native-**independent** logic (the `src/api` message bridge). Component rendering needs the native runtime, so those aren't unit-tested here.

**Continuous Native Generation:** `ios/` and `android/` are **generated by `npx expo prebuild`** from `app.config.js` and are **gitignored, not committed**. After a fresh `npm install`, run `npm run prebuild` before a native build. A native build needs Xcode / Android SDK + JDK 17 (set `JAVA_HOME` to a 17 JDK for gradle); release builds use EAS or a prebuild step in CI. There is **no longer any hand-maintained native code** — all native configuration (bundle id, name, permissions, ATS/cleartext, icon, splash) lives in `app.config.js`.

## Architecture

Entry chain: `index.ts` calls `registerRootComponent(App)` (from `expo`) -> `App.tsx`.

`App.tsx` is the router and the only stateful top-level component, wrapped in a `GestureHandlerRootView`. On mount it reads the AsyncStorage key `remote-control-url` (constant `STORAGE_KEY_RC_URL`); if unset it falls back to `DEFAULT_URL = 'https://spot.8x8.vc'`. Its `render`:

- A `react-native-drawer-layout` (`Drawer`) wrapping `RemoteControl` (`src/remote-control`, the WebView stack), with `src/settings-menu/SettingsMenu.tsx` as `renderDrawerContent`. The drawer is **controlled** via `state.menuOpen` (`onOpen`/`onClose` track swipe gestures; `_onResetApp` closes it). Drawer gestures use `react-native-gesture-handler` + `react-native-reanimated` (worklets babel plugin in `babel.config.js`).
- `src/setup.tsx` (`Setup`) overlaid only when `state.showSetup` is true.

App-level actions:
- **Reset** (`_onResetApp`): closes the drawer (`menuOpen: false`), calls `clearStorage()` down the WebView ref chain (which injects `localStorage.clear()`), removes the AsyncStorage URL key, then after a 1500ms timeout reloads or resets back to `DEFAULT_URL`.
- **Deployment** (`Setup`): validates an entered deployment URL by fetching `<url>/dist/config/config.js` and checking it contains `JitsiMeetSpotConfig` before persisting it via `_onSubmitEnteredUrl`.

WebView stack (`src/remote-control/`):
- `RemoteControl.tsx` owns the React Native `AppState` lifecycle (via `AppState.addEventListener` returning a subscription it `.remove()`s on unmount). While backgrounded it renders `LoadingScreen` instead of the WebView, and forwards `clearStorage()` / `reload()` to the inner ref only when active.
- `SpotWebview.tsx` is the actual `react-native-webview`. Key behavior:
  - Resolves the URL first via a `HEAD` `fetch` to follow redirects (`_resolveUrl`), rendering `LoadingScreen` until resolved.
  - Builds a custom user agent: `SpotController/1` in `__DEV__`, otherwise appended with app `name/version(buildNumber)` read from **`expo-application`** (`Application.applicationName` / `nativeApplicationVersion` / `nativeBuildVersion`). The `SpotController/1` token exists for feature detection by the web side.
  - `originWhitelist` is locked to the resolved host so non-Spot links open in the device browser, not the WebView.
  - Injects `_preventWebViewZoomScript` to force a non-zoomable viewport, sets `cacheEnabled={false}`, and renders a Retry error screen on load failure.
  - Keeps the screen awake via **`expo-keep-awake`** (`activateKeepAwakeAsync()` in `componentDidMount`, `deactivateKeepAwake()` in `componentWillUnmount`).
  - Registers the WebView ref with the message-bridge singleton (`api.setReference(ref)`) and wires `onMessage={api._onMessage}`.

JS<->web bridge (`src/api/index.ts`): a singleton `EventEmitter` (default export `api`).
- Web -> native: `_onMessage` parses `nativeEvent.data`, and for any payload with a `messageType` it re-emits `api.emit(messageType, messageData)`. Native code subscribes via `api.on(...)`.
- Native -> web: `_sendMessage(type, data)` calls `webViewRef.postMessage(JSON.stringify({ messageType, messageData }))`; `enterJoinCode()` is the one concrete helper, sending `connectWithCode`.

App metadata (name/version/buildNumber) comes from **`expo-application`** — there is no custom native `AppInfo` module anymore (the old `ios/.../AppInfo.m` + `android/.../AppInfoModule.java` were removed; expo-application reads the same `CFBundleDisplayName`/`versionName` etc.).

Supporting modules:
- `src/logger/` re-exports a `jitsi-meet-logger` instance (`getLogger('jitsi-meet-spot-controller', ...)`).
- `src/icons/` — an `Icon` component that renders an imported SVG as a fill-colored icon.
- `src/styles.ts` — shared `StyleSheet.create` fragments (full-screen black background via `StyleSheet.absoluteFill`, centered content, webView flex).
- `src/LoadingScreen.tsx` — full-screen `ActivityIndicator`, used as the loading/background placeholder throughout.

## Configuration (`app.config.js`)

Dynamic Expo config (the single source of native truth). Notable settings, all carried over from the old bare native projects: `name: 'Spot'`, bundle id / Android package `org.jitsi.spot` (overridable at build time via `SPOT_APP_ID`), `version` from `SPOT_VERSION` (default `2019.08.07`), `orientation: 'default'` (landscape + portrait), iOS `NSAppTransportSecurity.NSAllowsArbitraryLoads` + `NSBluetoothPeripheralUsageDescription`, Android `usesCleartextTraffic` (via the `expo-build-properties` plugin) and `allowBackup: false`, black background (`expo-system-ui`) and a black splash (`expo-splash-screen`). Icon/splash source is `assets/icon.png`.

## Conventions

- **Strict TypeScript.** `tsconfig.json` extends the repo base with `module: ESNext` / `moduleResolution: Bundler` (metro bundles), `jsx: react-native`, `lib: ["ESNext", "DOM"]`, and `types: ["node", "jest"]` (React/React-Native types come from the packages themselves — there is **no `@types/react-native`** with RN 0.85). **`noImplicitOverride` is off here** (class lifecycle overrides); the core `strict` family stays on. Relative imports are **extensionless** (bundler resolution).
- **Components** are class components typed as `React.Component<Props, State>` / `React.PureComponent<Props>`; prop typing is via **TypeScript prop interfaces** (e.g. `SpotWebviewProps`, reused by `RemoteControl`). Method handlers are `_`-prefixed and `.bind(this)` in the constructor. Refs use `React.createRef<T>()`.
- **Lint:** eslint flat config (`eslint.config.mjs`): `@eslint/js` + `typescript-eslint` recommended, with RN/jest globals. The React-specific eslint plugin is intentionally NOT used (TS covers prop typing).
- **Module layout:** feature folders (`remote-control/`, `settings-menu/`, `icons/`, `logger/`) each have an `index.ts` barrel that re-exports the implementation file; import from the folder, not the file.
- **SVG as components:** `metro.config.js` (Expo's `getDefaultConfig`, monorepo-aware via `watchFolders` + `nodeModulesPaths`) wires `react-native-svg-transformer/expo` so `.svg` files under `assets/icons/` are imported as React components. The `*.svg` module is typed (`React.FC<SvgProps>`) in `declarations.d.ts`. The Node tooling config files (`babel.config.js` = `babel-preset-expo` + `react-native-worklets/plugin`, `metro.config.js`, `jest.config.js`) and `app.config.js` stay CommonJS `.js` as Expo/metro expects.
- **Untyped deps** (the `jitsi-meet-logger` github dep) are declared in `declarations.d.ts`.
- The default deployment URL is hardcoded in `App.tsx` (`DEFAULT_URL`) — there is a `FIXME` to make it build-time configurable, so don't assume it is.
